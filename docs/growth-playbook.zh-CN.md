# 增长 Playbook

这份文档不是代码规范，而是 Parallel Worlds 作为开源项目持续增长的操作系统。

## 目标

把外部注意力稳定地转化成三样东西：

- GitHub stars
- 外部贡献者
- 可重复的公开推进节奏

Star 很重要，因为它直接影响项目被发现的概率。但 star 应该被看作“持续做对的公开工作”后的结果，而不是唯一目标本身。

## 增长原则

- 公开构建，不要长时间沉默后才一次性发布
- 既优化架构，也优化可演示性
- 每次发布都要比上一次更容易被理解
- 给程序员和非程序员都留出可参与的入口

## 每周要看的指标

每周至少记录一次：

- GitHub 总星标数
- 周新增星标
- README 浏览到星标的转化
- 外部用户新开的 issue 数
- 外部用户新开的 PR 数
- 可复用的 demo clip、截图、故事示例数量
- 从第一次访问到第一次本地跑通所需时间

## 四个增长飞轮

### 1. 产品飞轮

首次上手越顺滑，成功 demo 的概率越高；成功 demo 越多，越容易被分享；被分享越多，越容易获得 star。

当前优先项：

- 更顺滑的 setup
- 更强的 starter seeds
- 更清晰的生成状态表达
- 更容易分享的输出结果

### 2. 内容飞轮

每一个已发布改进，都应该能产出内容：

- 一条有明确 hook 的短帖
- 一个视觉素材，比如截图或短视频
- 一个面向开发者的技术角度
- 一个面向故事/游戏爱好者的叙事角度

### 3. 社区飞轮

外部注意力应该被转化成可见参与：

- 快速响应 issue
- 标记适合新人的任务
- 主动展示社区成员做出的世界和实验
- 尽量缩短 PR review 时间

### 4. 发布飞轮

高频且易理解的 release 会让项目显得“活着”：

- 持续维护 release draft
- 说明这次改了什么、为什么值得关心
- 用小步快跑的方式推动 roadmap 前进

## 每周执行节奏

### 周一

- 看一遍指标
- 选一个产品改进项和一个增长改进项
- 判断本周有哪些工作适合公开发帖

### 周中

- 发出一个看得见的改进
- 保存对应的截图、短视频或 demo seed
- 趁着上下文还在，先把 release note 草稿写出来

### 周五

- 发布一条本周总结
- 如果优先级有变化，就更新 roadmap 或 backlog
- 回复 issue 和 showcase

## 需要持续维护的发布资产

这些东西要长期保持可用：

- 一个够强的 README
- 一段 30 到 60 秒的产品 demo
- 3 到 5 个稳定可复现的 starter seeds
- 一段清楚解释“为什么做这个项目”的介绍
- 一份能体现项目在推进中的 roadmap
- 与当前产品状态一致的截图和社交预览图

## 除了写代码，还要做什么

- 写 release notes
- 持续发布 build-in-public 更新
- 收集并展示社区里好的作品
- 维护 issue 质量和 review 速度
- 固化项目定位，避免外界看不懂
- UI 变化之后同步更新 demo 资产

## 当前执行重点

- 让访客在 30 秒内理解项目在做什么
- 降低首次运行失败率
- 积累能反复用于传播的视觉和内容素材
- 给外部贡献者更明确的起点

## 任务选择的判断标准

两个任务冲突时，优先做同时改善以下至少两项的那个：

- 更容易 demo
- 更容易运行
- 更容易分享
- 更容易贡献

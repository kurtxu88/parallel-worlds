# Growth Playbook

This is the operating system for growing Parallel Worlds as an open-source project, not just as a codebase.

## Goal

Turn curiosity into three things:

- stars
- contributors
- repeatable public momentum

Stars matter because they are discovery fuel. But stars should be treated as an outcome of useful public work, not the work itself.

## Growth Principles

- ship in public, not in batches of silence
- optimize for demoability as much as architecture
- make every release easier to understand than the last one
- create contribution paths for both coders and non-coders

## Metrics To Watch

Track these every week:

- total GitHub stars
- star growth by week
- README views to stars ratio
- external issues opened
- external pull requests opened
- number of reusable demo clips, screenshots, and story examples
- time from first visit to first successful local run

## The Four Growth Loops

### 1. Product loop

Better first-run experience leads to more successful demos, which leads to more shares, which leads to more stars.

Current priorities:

- smoother setup
- stronger starter seeds
- clearer story generation states
- more shareable outputs

### 2. Content loop

Each shipped improvement should produce content:

- one short post with a strong hook
- one visual artifact such as a screenshot or clip
- one technical angle for builders
- one narrative angle for story/game people

### 3. Community loop

Attention should be converted into visible participation:

- answer issues quickly
- label newcomer-friendly work
- celebrate showcases and experiments
- keep PR review time short

### 4. Release loop

Frequent, understandable releases make the project feel alive:

- draft release notes continuously
- summarize what changed and why it matters
- keep the roadmap moving in small visible steps

## Weekly Operating Rhythm

### Monday

- review metrics
- pick one product improvement and one growth improvement
- identify what can become a public post later in the week

### Midweek

- ship a visible improvement
- capture a screenshot, clip, or demo seed
- draft the release note while context is fresh

### Friday

- publish a short recap
- update roadmap or backlog if priorities changed
- respond to issues and showcases

## Launch Assets To Maintain

Always keep these ready:

- a strong README
- a 30 to 60 second product demo
- 3 to 5 reliable starter seeds
- a one-paragraph explanation of why the project exists
- a roadmap that feels active, not aspirational
- screenshots or social cards that still match the current product

## What To Do Besides Coding

- write release notes
- publish build-in-public updates
- collect and feature good community outputs
- improve issue hygiene and review speed
- document positioning so the project stays legible
- keep demo assets fresh after UI changes

## Current Execution Priorities

- increase the percentage of visitors who can understand the project in under 30 seconds
- reduce first-run failure points
- create more assets that can be posted repeatedly without sounding repetitive
- give outside contributors clearer places to start

## Decision Filter

When choosing between two tasks, prefer the one that improves at least two of these:

- easier to demo
- easier to run
- easier to share
- easier to contribute to
